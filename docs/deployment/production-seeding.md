# Production SQL Data Transfer (Forge)

This guide documents the safe path to migrate alert data from a local dataset
into production using SQL dumps generated by `db:export-sql`.

## Scope

This process only covers alert-source tables exported by `db:export-sql`:

- `fire_incidents`
- `police_calls`
- `transit_alerts`
- `go_transit_alerts`

## Prerequisites

1. Local branch is up to date and clean enough to inspect generated seeder diffs.
2. Production schema is current (`php artisan migrate --force` must run before seeding).
3. You have a production DB backup/snapshot you can restore quickly.
4. You understand generated seeders may contain operationally sensitive location text and incident details.

## Step 1: Generate SQL Export Locally

Run from repository root:

```bash
./scripts/export-alert-data.sh --sail
```

Optional tuning:

```bash
./scripts/export-alert-data.sh --sail --output=/tmp/alert-export.sql --chunk=500 --compress
```

Expected behavior:

1. Runs `db:export-sql`.
2. Confirms export file exists.
3. Prints SHA-256 and transfer guidance.
4. If `--compress` is used, output is `.sql.gz`.

## Step 1.5: Run Final Quality Gate Verification

Before promoting the export workflow, run track quality verification:

```bash
php artisan test --filter=ExportAlertDataSqlTest
php artisan test --filter=ImportAlertDataSqlTest
./vendor/bin/sail artisan test --coverage --min=90
```

The command-level tests validate SQL generation/import guardrails and the
coverage command validates quality thresholds for the new modules.

## Step 2: Transfer SQL File To Destination

Use an out-of-band transfer path (SCP/SFTP/Forge file manager). Do not commit
SQL exports to git.

Examples:

```bash
scp /tmp/alert-export.sql forge@server:/home/forge/<your-site>/storage/app/
```

## Step 3: Run Import In Laravel Forge

### Option A: One-off command

In Forge, open the site server and run:

```bash
cd /home/forge/<your-site>
php artisan migrate --force
php artisan db:import-sql --file=storage/app/alert-export.sql --force
```

If the transferred file is compressed:

```bash
gunzip -c storage/app/alert-export.sql.gz > storage/app/alert-export.sql
php artisan db:import-sql --file=storage/app/alert-export.sql --force
```

### Option B: Pipe directly to `psql`

For very large files, stream decompression directly:

```bash
gunzip -c storage/app/alert-export.sql.gz \
  | PGPASSWORD="$DB_PASSWORD" psql -h "$DB_HOST" -p "$DB_PORT" -U "$DB_USERNAME" -d "$DB_DATABASE"
```

`db:import-sql` rejects `.gz` files intentionally, so decompress before passing
`--file`.

## Step 4: Post-Deploy Validation

1. Check row counts for all four alert tables.
2. Confirm dashboard feed renders historical and active incidents.
3. Spot-check that `created_at`/`updated_at` values match source records.
4. Re-run the import command once to confirm idempotency (`ON CONFLICT (id) DO NOTHING` prevents duplicates).

## Legacy Seeder Workflow (Deprecated)

The seeder-based workflow remains available for compatibility only:

- `php artisan db:export-to-seeder`
- `php artisan db:verify-production-seed`
- `./scripts/generate-production-seed.sh`

Use the SQL export/import workflow for all new production data transfer operations.

## Security Warnings

- Do not run this process against the wrong environment.
- Keep SQL export files out of version control.
- Never include credentials or tokens in shell history when using `psql`.
- Run DB backup/snapshot before import in production.

## Troubleshooting

| Symptom | Cause | Resolution |
|---|---|---|
| `psql binary is required` | Destination host is missing PostgreSQL client tools | Install `psql` on host or run import through Sail in an environment that includes `psql`. |
| `Gzip-compressed SQL files are not supported` | `.sql.gz` passed directly to `db:import-sql` | Decompress to `.sql` first, then rerun import. |
| `Refusing to import into testing database` | Guardrail triggered for `APP_ENV=testing` or `_testing` DB | Use a non-testing target or pass `--allow-testing` for controlled test runs. |
| Import runs but data is incomplete | Export ran on stale/empty source DB | Refresh source data, rerun export, and verify counts before transfer. |
| Import fails with schema errors | Production migrations not applied | Run `php artisan migrate --force` before import. |
