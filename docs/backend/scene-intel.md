# Scene Intel System

This document describes the architecture, schema, synthetic generation logic, and API surface for the Scene Intel feature.

## Overview

Scene Intel provides real-time operational updates for fire incidents. The system:
- Automatically generates "synthetic intel" by diffing Toronto Fire CAD feed snapshots
- Allows authorized users to add manual notes
- Delivers updates via polling-based API (WebSocket support deferred)
- Embeds latest updates in main feed for reduced latency

Synthetic intel captures:
- **Alarm level changes** (escalation/de-escalation)
- **Resource status updates** (units dispatched/cleared)
- **Phase changes** (incident closure)

## Architecture

```
Toronto Fire CAD Feed (every 5 minutes)
    ↓
FetchFireIncidentsCommand
    ↓
Capture previous state + Update/Create incident
    ↓
SceneIntelProcessor.processIncidentUpdate(incident, previousData)
    ↓
Generate IncidentUpdate records (ALARM_CHANGE, RESOURCE_STATUS, PHASE_CHANGE)
    ↓
IncidentUpdate persisted to database
    ↓
Frontend polls GET /api/incidents/{eventNum}/intel
    ↓
SceneIntelTimeline renders updates
```

## Database Schema

### `incident_updates`

| Column | Type | Description |
|---|---|---|
| `id` | bigint (PK) | Auto-increment primary key |
| `event_num` | string (FK) | Foreign key to `fire_incidents.event_num` (ON DELETE CASCADE) |
| `update_type` | string | Enum: `milestone`, `resource_status`, `alarm_change`, `phase_change`, `manual_note` |
| `content` | text | Human-readable update description |
| `metadata` | json | Previous values, unit codes, user ID for manual entries |
| `source` | string | Enum: `synthetic`, `manual` |
| `created_by` | bigint (FK, nullable) | Foreign key to `users.id` (for manual entries) |
| `created_at` | timestamp | Creation time |
| `updated_at` | timestamp | Last update time |

**Indexes:**
- Primary key on `id`
- Composite index on `(event_num, created_at)` for timeline queries
- Index on `update_type`
- Index on `created_at`

**Model:** `App\Models\IncidentUpdate`

**Factory:** `Database\Factories\IncidentUpdateFactory`

**Enum:** `App\Enums\IncidentUpdateType` (`MILESTONE`, `RESOURCE_STATUS`, `ALARM_CHANGE`, `PHASE_CHANGE`, `MANUAL_NOTE`)

## Synthetic Intel Generation

### SceneIntelProcessor

Service: `App\Services\SceneIntel\SceneIntelProcessor`

Method: `processIncidentUpdate(FireIncident $incident, ?array $previousData): void`

The processor compares new incident state against previous state to detect changes:

#### 1. Alarm Level Changes (`ALARM_CHANGE`)

Detects when `alarm_level` increases or decreases:

```php
// Example: alarm_level 1 → 2
Content: "Alarm level increased from 1 to 2"
Metadata: {"previous_level": 1, "new_level": 2}
```

```php
// Example: alarm_level 2 → 1
Content: "Alarm level decreased from 2 to 1"
Metadata: {"previous_level": 2, "new_level": 1}
```

#### 2. Resource Status Updates (`RESOURCE_STATUS`)

Compares `units_dispatched` (comma-separated string) to detect:

**Units Dispatched:**
```php
// Example: Previous units "E123,E456" → New units "E123,E456,E789"
Content: "Unit E789 dispatched"
Metadata: {"unit": "E789", "action": "dispatched"}
```

**Units Cleared:**
```php
// Example: Previous units "E123,E456,E789" → New units "E123,E789"
Content: "Unit E456 cleared from scene"
Metadata: {"unit": "E456", "action": "cleared"}
```

#### 3. Phase Changes (`PHASE_CHANGE`)

Detects incident closure when `is_active` transitions from `true` to `false`:

```php
Content: "Incident marked as resolved"
Metadata: {"previous_status": "active", "new_status": "resolved"}
```

**Special Case: Bulk Deactivation**
- The command bulk-updates missing incidents via `whereNotIn()`
- Closure intel is generated by explicitly processing the deactivated set
- Jitter guard prevents duplicate closures if incident reactivates

### Integration Point

Command: `App\Console\Commands\FetchFireIncidentsCommand`

The processor is invoked:
1. **After `updateOrCreate`** - for alarm/resource changes
2. **After bulk deactivation** - for closure intel

Scheduled in `routes/console.php` to run every 5 minutes.

### Resilience & Failure Modes

- **Non-blocking per-incident failures:** Exceptions thrown by `SceneIntelProcessor::processIncidentUpdate()` are caught per incident and logged as warnings. Ingestion continues for the rest of the batch.
- **Failure rate monitoring:** If more than 50% of Scene Intel attempts fail in a run, a warning is logged and emitted to the console to surface systemic issues.
- **Job retry policy:** `FetchFireIncidentsJob` retries the command up to 3 times with a 30-second backoff. Note that Scene Intel exceptions are intentionally swallowed and **do not** trigger a job retry.
- **Acceptable partial failure:** Occasional Scene Intel failures are tolerated; repeated warnings or sustained >50% failure rate indicate data or database issues that should be investigated.

## API Endpoints

### GET `/api/incidents/{eventNum}/intel`

**Description:** Retrieve full chronological timeline for an incident

**Authentication:** None (public read-only)

**Response:**
```json
{
  "data": [
    {
      "id": 42,
      "update_type": "alarm_change",
      "content": "Alarm level increased from 1 to 2",
      "source": "synthetic",
      "created_at": "2026-02-13T14:32:00-05:00",
      "metadata": {
        "previous_level": 1,
        "new_level": 2
      }
    }
  ],
  "meta": {
    "event_num": "F26012345"
  }
}
```

**Controller:** `App\Http\Controllers\SceneIntelController::timeline()`

**Resource:** `App\Http\Resources\IncidentUpdateResource`

### POST `/api/incidents/{eventNum}/intel`

**Description:** Add manual intel entry (authorized users only)

**Authentication:** Required (`auth`, `verified` middleware)

**Authorization:** Gate `scene-intel.create-manual-entry` (defined in `AppServiceProvider`)

**Request:**
```json
{
  "content": "Fire is under control"
}
```

**Validation:** `App\Http\Requests\StoreSceneIntelEntryRequest`
- `content`: required, string, max 1000 characters
- Content is sanitized with `strip_tags()` before validation to prevent XSS

**Response:**
```json
{
  "data": {
    "id": 43,
    "update_type": "manual_note",
    "content": "Fire is under control",
    "source": "manual",
    "created_at": "2026-02-13T14:35:00-05:00",
    "metadata": {
      "created_by": 1
    }
  }
}
```

**Controller:** `App\Http\Controllers\SceneIntelController::store()`

## Authorization

### Gate: `scene-intel.create-manual-entry`

Defined in: `App\Providers\AppServiceProvider::configureSceneIntelAuthorization()`

**Policy (Fail-Closed):**
1. User must have verified email
2. `allowed_emails` config array must be non-empty
3. User email must be in allowed list

**Configuration:** `config/scene_intel.php`
```php
return [
    'allowed_emails' => env('SCENE_INTEL_ALLOWED_EMAILS', ''),
];
```

Environment variable: `SCENE_INTEL_ALLOWED_EMAILS` (comma-separated)

**Default behavior:** Denies all manual entries (fail-closed security model)

## Optimization: Embedded Intel Summary

### FireAlertSelectProvider Enhancement

To reduce blank state before first poll, `FireAlertSelectProvider` embeds latest updates:

**Added Columns:**
- `intel_summary` - JSON array of latest 3 updates
- `intel_last_updated` - Timestamp of most recent update

**Implementation:**
- Efficient subquery fetches latest 3 updates per incident
- Tri-driver support: SQLite uses `json_group_array`/`json_object`, PostgreSQL uses `json_agg`/`json_build_object`, MySQL uses `JSON_ARRAYAGG`/`JSON_OBJECT`
- Police/Transit providers include `NULL` placeholders for `UNION` compatibility

**Query Pattern (PostgreSQL):**
```sql
SELECT
  -- ... standard columns ...
  (
    SELECT COALESCE(json_agg(
      json_build_object('id', t.id, 'type', t.update_type, ...)
      ORDER BY t.created_at DESC, t.id DESC
    ), '[]'::json)
    FROM (
      SELECT id, update_type, content, created_at, metadata
      FROM incident_updates
      WHERE incident_updates.event_num = fire_incidents.event_num
      ORDER BY created_at DESC
      LIMIT 3
    ) as t
  ) as intel_summary,
  (
    SELECT to_char(MAX(created_at), 'YYYY-MM-DD"T"HH24:MI:SS"Z"')
    FROM incident_updates
    WHERE incident_updates.event_num = fire_incidents.event_num
  ) as intel_last_updated
```

See `app/Services/Alerts/Providers/FireAlertSelectProvider.php` for the SQLite and MySQL equivalents.

- `manual_note` - Gray badge with note icon

### useSceneIntel Hook

Hook: `resources/js/features/gta-alerts/hooks/useSceneIntel.ts`

**Behavior:**
- Fetches `/api/incidents/{eventNum}/intel`
- Validates response with `SceneIntelResponseSchema` (Zod)
- Polls every 30 seconds when component is mounted
- Stops polling when unmounted
- Returns: `{ data, isLoading, error, refresh }`

**Error Handling:**
- Schema validation failures logged to console
- Network errors exposed via `error` state
- Component renders gracefully with empty state

## Maintenance & Pruning

**Policy:** Incident updates older than 90 days are permanently deleted.

**Implementation:** Model uses `Illuminate\Database\Eloquent\MassPrunable` trait

**Command:** Scheduled via `php artisan model:prune --model="App\Models\IncidentUpdate"`

**Schedule:** Daily at `00:00` in `routes/console.php`

**Tests:** `tests/Feature/SceneIntel/IncidentUpdatePruningTest.php`
