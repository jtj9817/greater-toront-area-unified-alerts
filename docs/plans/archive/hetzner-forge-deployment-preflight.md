## Critical Blockers (Must Resolve Before First Deploy)

### 1. Core Application Environment Variables ‚Äî ‚öÝ Must Be Set in Forge

Set these in Forge under **Environment ‚Üí Edit Environment**:

```env
APP_NAME="GTA Alerts"
APP_ENV=production
APP_DEBUG=false
APP_URL=https://your-domain.com
APP_KEY=                          # Generated by Forge or `php artisan key:generate`

# PostgreSQL (recommended for production)
DB_CONNECTION=pgsql
DB_HOST=127.0.0.1
DB_PORT=5432
DB_DATABASE=gta_alerts
DB_USERNAME=forge
DB_PASSWORD=<forge-generated>

# MySQL (alternative, if Forge provisions MySQL instead)
# DB_CONNECTION=mysql
# DB_HOST=127.0.0.1
# DB_PORT=3306
# DB_DATABASE=gta_alerts
# DB_USERNAME=forge
# DB_PASSWORD=<forge-generated>

QUEUE_CONNECTION=database         # Or redis (see Redis section below)
CACHE_STORE=database              # Or redis
SESSION_DRIVER=database

LOG_CHANNEL=stack
LOG_STACK=daily
LOG_LEVEL=warning
```

**Review every value** ‚Äî do not copy `.env` from development. Forge generates the database credentials; use those.

> **Note:** PostgreSQL is the production-tested database driver. All four alert providers (`FireAlertSelectProvider`, `PoliceAlertSelectProvider`, `TransitAlertSelectProvider`, `GoTransitAlertSelectProvider`) implement explicit PostgreSQL branches using `json_build_object`, `json_agg`, `to_tsvector`/`plainto_tsquery` FTS, and `ILIKE` substring fallback. GIN full-text indexes are created by migration `2026_02_25_000000_add_pgsql_fulltext_indexes_to_alert_tables.php` using `CREATE INDEX CONCURRENTLY` (non-blocking). MySQL continues to work for local development (`phpunit.mysql.xml`).

### 2. Queue Worker ‚Äî Must Add Forge Daemon

The following jobs run via the queue and are critical to application function:

- Feed ingestion jobs (Fire, Police, TTC, GO Transit)
- Notification delivery (`DeliverAlertNotificationJob`, `FanOutAlertNotificationsJob`)
- Daily digest (`GenerateDailyDigestJob`)

**Add a Daemon in Forge ‚Üí Daemons:**

```
Command:   php artisan queue:work --queue=default --tries=3 --sleep=3 --max-time=3600
Directory: /home/forge/your-site
User:      forge
Processes: 2
```

Without this, queued jobs will pile up indefinitely and feeds will never sync.

---

### 3. Scheduler (Cron) ‚Äî Must Add Forge Cron Entry

The scheduler manages all timed data fetches and maintenance tasks. **Add in Forge ‚Üí Scheduler:**

```
Command:   php artisan schedule:run
User:      forge
Frequency: Every Minute
```

Forge translates this to the standard Laravel cron entry:
```
* * * * * cd /home/forge/your-site && php artisan schedule:run >> /dev/null 2>&1
```

**Scheduled tasks that require this:**

| Task | Frequency | Overlap Guard |
|---|---|---|
| `fire:fetch-incidents` | Every 5 min | 10 min max overlap lock |
| `police:fetch-calls` | Every 10 min | 10 min max overlap lock |
| `transit:fetch-alerts` | Every 5 min | 10 min max overlap lock |
| `go-transit:fetch-alerts` | Every 5 min | 10 min max overlap lock |
| `notifications:prune` | Daily | ‚Äî |
| `queue:prune-failed` | Daily | ‚Äî |
| `model:prune` (IncidentUpdate) | Daily | ‚Äî |
| `monitor:queue-depth` | Every 5 min | ‚Äî |

## High Priority (Configure Before Going Live)

### 4. Broadcasting / Real-Time Notifications ‚Äî ‚öÝ Needs Provider Choice

Notifications broadcast over WebSockets on a private channel (`private-users.{userId}.notifications`). The current default is `BROADCAST_CONNECTION=log` (dev-only no-op).

**Option A ‚Äî Laravel Reverb (recommended, self-hosted)**

Reverb runs as a long-running process on the same server. Add a **Daemon in Forge:**
```
Command: php artisan reverb:start --port=8080
```

Set environment variables:
```env
BROADCAST_CONNECTION=reverb
REVERB_APP_ID=gta-alerts
REVERB_APP_KEY=<random-string>
REVERB_APP_SECRET=<random-string>
REVERB_HOST=your-domain.com
REVERB_PORT=443
REVERB_SCHEME=https

VITE_REVERB_APP_KEY="${REVERB_APP_KEY}"
VITE_REVERB_HOST="${REVERB_HOST}"
VITE_REVERB_PORT="${REVERB_PORT}"
VITE_REVERB_SCHEME="${REVERB_SCHEME}"
```

**Option B ‚Äî Pusher (managed SaaS, simpler setup)**
```env
BROADCAST_CONNECTION=pusher
PUSHER_APP_ID=...
PUSHER_APP_KEY=...
PUSHER_APP_SECRET=...
PUSHER_APP_CLUSTER=mt1

VITE_PUSHER_APP_KEY="${PUSHER_APP_KEY}"
VITE_PUSHER_APP_CLUSTER="${PUSHER_APP_CLUSTER}"
```

**If real-time notifications are not needed for launch:** leave `BROADCAST_CONNECTION=log` and add broadcasting later. The application will work without it; notifications simply will not push to open browser tabs.

### 6. Security Headers ‚Äî ‚öÝ Not Configured in Codebase

The codebase has solid application-level security (CSRF, rate limiting, SQL parameterisation, no `APP_DEBUG` in production) but does not set HTTP security headers. Add these to the Forge Nginx config (**Edit Nginx Config** in Forge):

```nginx
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;
```

Place these inside the `server {}` block.

## Recommended Improvements (Not Blockers)

### 7. Redis ‚Äî Upgrade from Database Driver

The application defaults to the `database` driver for both queue and cache. This works but adds read/write load to MySQL under sustained use. Redis is already present in the development Docker Compose stack.

**To enable on the Hetzner VPS:**

Install Redis on the server (Forge can provision it, or install manually):
```bash
sudo apt install redis-server
```

Update environment:
```env
REDIS_HOST=127.0.0.1
REDIS_PASSWORD=null
REDIS_PORT=6379

QUEUE_CONNECTION=redis
CACHE_STORE=redis
SESSION_DRIVER=database   # Keep sessions in DB for simplicity
```

This is optional for initial launch but recommended if the queue worker processes more than a few hundred jobs per hour.

## Deployment Script

We have added a custom deployment script in version control that correctly compiles the React/Inertia frontend using `pnpm`.

Configure this as your **Forge Deploy Script** (Site ‚Üí Deployment ‚Üí Deploy Script):

```bash
cd /home/forge/your-site
git pull origin $FORGE_SITE_BRANCH

# Delegate to the version-controlled script
bash scripts/forge-deploy.sh
```

This ensures that Forge will install `pnpm` dependencies, build the React components, run migrations, and safely reload PHP and the queue workers.

> **Note:** Ensure `pnpm` is installed globally on your Forge server (e.g., `npm install -g pnpm`) if it isn't already.

## Known Non-Issues (Audit Findings That Are Fine As-Is)

- **SQLite in development:** The codebase uses SQLite as the default test/dev fallback (no native FTS; outer-query `LIKE` is used instead). No action needed.
- **No Horizon/Telescope installed:** Not required. The queue depth monitor cron job and failed job logging provide sufficient observability without adding infrastructure complexity.
- **No custom Nginx config in repo:** Intentional ‚Äî Forge manages its own Nginx configuration and site templates. The `.htaccess` in `public/` handles Apache fallback but is not used under Nginx/Forge.
- **Docker/Sail not used in production:** Correct. Forge manages PHP-FPM, Nginx, and MySQL natively. Sail is development-only.
- **Broadcast connection defaults to `log`:** Safe default. Broadcasting is an enhancement; the application is fully functional without it.
- **`DB::prohibitDestructiveCommands(app()->isProduction())`** ‚Äî already in `AppServiceProvider`. This will prevent accidental `migrate:fresh` or `db:wipe` in production.

## Deferred / Known Limitations

### Mail / Transactional Email ‚Äî ‚öÝ Skipped for Launch

The application currently relies on in-app notifications and does not use email for core features. Password reset functionality will be unavailable.

**Configuration in Forge:**
Keep the default `log` driver to prevent errors if the app attempts to send mail.

```env
MAIL_MAILER=log
```

**Impact:**
- **Password Reset:** Will not work. Users forgetting passwords will need admin intervention.
- **Email Verification:** Will not work (but `MustVerifyEmail` is currently disabled in `User.php`).
- **Daily Digest:** Will be generated as an in-app notification only.
